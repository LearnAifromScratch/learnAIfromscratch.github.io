What is this

How to game the current technical interview system for fun and profit. We need to be able to do this: "it's not enough to be able to solve the problems, you need to be able to solve them on the spot, without much thinking, in an interview context". Codility a popular filtering site for testing applicants based it's service on the IOI competition, where all problems usually have an easy but 'suboptimal' solution, and the optimal solution requires algorithmic skill, so let's get that skill.

Learning procedure

  • Learn a concept in the book/course
  • Try some problems
  • Design simple software out of what we just learned, release it with full documentation

The software you make can be anything like a visualization of the algorithm, forking and modifying some open source software and changing out algorithms, a ridiculously efficient log searching tool, doesn't matter just anything you can point to and say I built that and it has tests/documentation the objective is to build a portfolio to show off.

One example is all the string manipulation algorithms we'll learn, think of all the plugins for VSCode or other dev tools you could modify with those algorithms (or write your own).

C++ crash course

C++ is a generic programming language not really an 'object oriented language' if you look at the STL or standard template library. You can alter all the built-ins by supplying custom functions without needing to manipulate function pointers. C++ can pass functions as arguments to another function and return new functions so functional programming is possible. You can make immutable vectors and lists, use filter/fold/map(std::transform)/reduce from the STL, hack a template together to do lazy eval, partial function application and currying, it's awkward but works.

OCaml crash course

Here's a quick rundown how abstract OCaml can get 

# let compose f g = fun x -> f(g x);;
val compose : ('a -> 'b) -> ('c -> 'a) -> 'c -> 'b = <fun>

# compose (fun x -> x + x) (fun y -> y + y) 2;;
- : int = 8

Above is a function composition f(g(input)) where a function g is applied first to an input and the result is the input for function f. Lambda syntax (a function without a name) in OCaml is fun x -> x and you can bind that anonymous function to a name with let. 

val compose : ('a -> 'b) -> ('c -> 'a) -> 'c -> 'b = <fun>

These arrows form typ -> typ' where typ is the domain type (the type of the inputs) and typ' is its range type (the type of the result). The last -> 'b on the right is the result, and everything else to the left denotes the parameters that function takes. Arrows with brackets around them (typ -> typ') denote a function is expected as a parameter. These are all type variables meaning they change to whatever data you give it, if you give it strings it's now type string. They are called alpha 'a, beta 'b and gamma 'c. Think of them like basic math variables: a + b = 2 where a and b can be the same value 1 or different like -2 and 4, it's the same deal with type variables 'a -> 'b can be int -> int or int -> string but should be consistent or you will get a type error. The = <fun> notation is a placeholder for the byte code generated by the top-level since it would make no sense to display the byte code in the top-level. 

let g x =
  let f = fun y -> y + x in
  let x = 1 in
  f x

OCaml developers often use nested evaluation sometimes called inlining which is a way to both limit scope like importing a library who's namespace can't escape the let binding and for compiler optimization as you give it all it needs at once like Common Table Expressions in SQL 'here's everything I want to do as a snippet, you figure out how best to do it'.

 let f = fun y -> y + x in

The step (fun y -> y + 10) 3 is similar to the lambda calculus syntax: function on the left applied to value on the right and evaluated via substitution until you reach a value. # let derivative dx f = function x -> (f(x +. dx) -. f(x)) /. dx;;
val derivative : float -> (float -> float) -> float -> float = <fun>

The weird dot notation +. or /. denotes a float number is expected. The type arrow notation denotes a function that takes a float and a function as input, and returns a function from float -> float.

This calculates the derivative of an arbitrary function f. Imagine a power function f(x) = x2 and you want it's derivative, which uses the formula f(x)' = nxn-1 where you bring down the exponent, subtract 1 from the original exponent, and multiply or f(x) = 2x1 with the derivative dx f above function being the longer version of that to calculate any derivative not just a power function. 

# let squared = derivative 1e-5 (fun x -> x *. x);;
val squared : float -> float = <fun>
# squared 4.;;
- : float = 8.00000999952033

# let three = derivative 1e-5 (fun x -> x *. x *. x);;
val three : float -> float = <fun>
# three 2.;;
- : float = 12.0000600002612128

I passed derivative a tiny dx, and a power function which is f x = x2, it returned the derivative of f x = x2 which in calculus is another function f x = 2x. I applied that to 4 and got 8. The three derivative is f x = x3 or using the power derivative formula it's f x = 3x2 and I applied that to 2 and got 12 as expected.

Now we can go crazy: 

# let pi = 4.0 *. atan 1.0;;
val pi : float = 3.14159265358979312

# let sin''' = (power (derivative 1e-5) 3) sin;;
val sin''' : float -> float = <fun>

# sin''' pi;;
- : float = 0.999998972517346263

Note that 'sin' is a built-in OCaml function. What happened here was I created pi, a constant value. Then I calculated the third derivative of sine by computing the cubic power of the derivative function, which returns a new function, and applying that to sin. I could then apply that final returned function to pi and get back a number. 

# let sin''' = (power (derivative 1e-5) 3) sin
   in sin''' pi;;
- : float = 0.999998972517346263

And finally as a common idiom in OCaml combining everything into one block which reads: let variable sin''' be the function that is returned from (power (derivative 1e-5) 3), apply that returned function to sin (built-in) and then apply whatever is returned to pi in sin''' pi or (function) pi.

Algorithm analysis

If you didn't take CS19 I will summarize.

You want to know the running time of your program, so you use an abstract cost model to convert source code to resources required to run that source. Running time of a program is usually dependent on the size of it's input, so if there's a dependency then we can use a math function to model it.

You use the cost model to manually go through the source and assign costs, the details of how you do this doesn't matter because we'll be taught a specific model in the Advanced Data Structures course. This is f(x) so given an input of size x, returns the running time cost of that input.

To describe the growth of this f(x) running time over a huge amount of inputs, we want a second function g(x) where we arbitrarily choose well known math functions that we can use to provide a bound on the growth of f(x). To recap f(x) is something you derive from your cost model and program source code and outputs running time, and g(x) is a function you choose that best describes a bound of the growth of f(x) so you could say the output of g(x) is a growth model of f(x).

A simplified definition of Big-O is therefore:

\[f(x) \le c \cdot g(x)\]

Meaning for all inputs to f(x), the running time is bounded to whatever you chose g(x) to be and a constant factor c which gives us some relaxed wiggle room to ignore constants in Big-O because we only care about polynomial time and exponential time. If a program is in polynomial time like n2 then we can use it or optimize it to something better, while you almost never can use an exponential running time 2x on inputs larger than small sizes.

And that's it. The rest are details we'll cover as we go but somehow in every intro course except CS19 this is never explained they immediately get students confused by throwing around math notation and graphs.

Amortization

This couse uses the potential method of amortized analysis, which we will learn now. The reason we have skew heaps and all sorts of weird trees is because in the 80s they discovered amortized analysis, and modified dozens of data structures.

Watching his 6.046 lecture here.

He begins using an example of hash table (array) doubling. We already did hashing in CS19 where the idea is to compute efficiently on the keys in a balanced binary search tree and each key maps to an array index. The array (table) is size m and the number of keys you insert is n. When the array fills up you want to increase the table size, so you double the size of m and copy over all the old array data and add the new insertion n. Then when double m is full double again. When he says 'you only double log n times' and proceeds to demonstrate a growing geometric series of doubling recall that 2log n is n. Suppose m = 1. When you have 2 keys you double to 21, 3 keys you double to 22, 5 keys you double to 23. If you stop at 8 keys, you have doubled 3 times, or log(8) = 3 or log(n) times.

He explains 2-3 trees later but we don't need to know what a 2-3 tree is to understand the example just the interface that insertions/deletions cost O(log n). When he was adding up the total costs he ignored constants and reasoned deletion was at most 2x the inserts so 2i(log(n)) which is O(i(log(n)). Deletions were free here because the total cost ended up being O(c + i(log(n))) and when you divide up that total over all operations deletions end up 'costing nothing' in amortized costs.

@41:40 this is the method we're interested in, the potential method. Note that we are using it per operation, so amortizing the cost of incrementing +1 operation or inserting a new node operation. After enough examples of 2-3 trees and 5-2 trees we see that anything that ends up in a 'bad state' where the next operation generates a cost such as splitting or merging is what you use as your potential function phi(end state) or phi(after op). I guess the intuition here is that the potential function builds up cost energy with every operation, and the goal is to drain that cost by subtracting the expensive end state from the beginning state of an operation and hopefully getting a constant.

The formula: actual cost of op + phi(after op) - phi(before op)

In Tarjan's original paper, he uses an example of a stack operation, defining the potential function of a stack to be the number of items it contains, which is of course non-negative as required. If we have a stack with n amount of items, and we perform a sequence of operations that does k pops (removals) and 1 push, then we have

  • actual cost: O(k) + O(1)
  • before op: n (amount of stack items already there)
  • after op: n - k + 1 (the new total n)
  • Total + after - before:
    • (k + 1) + (n - k + 1) - n = 2.

The k pops cancel out, n cancels out, amortized cost of that sequence is constant. This is something we have to see by example over and over to really get it which we will in 6.851
