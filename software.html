Strategy

Find a mentor. There is someone somewhere doing what you want to do and making a lot of money doing it and your task is to do enough material here to reach a level where you can get in with them somehow and absorb everything. Once you are in all you have to do is be the reliable person they turn to whenever some (tedious at first) work needs to be done and if this work over time becomes increasingly more skilled then you have found the right mentor and you are set for life. If they get promoted you get promoted. If they go off and found a new company you are the first hire. If they get hired at a top tier trading firm everyone says you have no chance of getting into guess who they will tell the hiring managers they should also hire.

You approach this person and sell yourself telling them you are available if they have some work because you want to learn it all from the ground up. There is always something to do and your goal is to be this magic portal where tedious work they despise doing can be dumped and it returns as per specifications they gave you. If you can do this then there's time for them to show you things. You eventually make your way up to highly sophisticated work until you end up being essential to them where they couldn't imagine going back to working without you because it's a mutually beneficial relationship. I actually took all the above advice from a comedian many years ago and it worked. In finance this can be tricky because of signing multiple NDAs even between internal departments but if you sell yourself to them that you are serious about being mentored they will tell management to make it happen.

USACO guide

In competitive programming you submit source code as a solution to a problem and the online judge compiles and runs your code against tests. There is a huge list of online judges available like Kattis, SPOJ, Timus, Aizu, and more worldwide.

First task is to choose one of these languages. We'll learn C++ for USACO so pick something else here that you want to use. List all the problems by easiest and start with the very first one 'Velkomin!'. You get unlimited submissions to the Kattis judge.

The point of doing this is to learn algorithm design which really is 'system design' there's some complex thing you want to build that needs to be optimized as much as possible and this is how you get better at recognizing patterns and matching them to typical algorithm design strategies.

Input/Output

Read the book or online documemtation of the language you chose or use the USACO guide first chapter to teach yourself how input/output works in C/C++/Java/Python. There's the USACO online IDE too if you want.

The problem qaly is an example of how to take multiple lines of input. Many of these problems the first input will be how much more input to expect which you can then loop on: 

#include <iostream>
using namespace std;
int main(){
    int tt;  
    double sum = 0;  
    cin >> tt;   // pipe first stdin to tt
    while (tt--) {  // false when input empty 
        double a, b;  // each loop declare these 2 new assignables
        cin >> a >> b; // take 2 more inputs
        sum = sum + (a * b);  // accumulator
        }
    cout << sum;  // pipe sum to stdout
}

Finish up to 1.4 Easy problems on Kattis or until you get how I/O works in your language.

If you took the accelerated intro course then generic programming is the same as 'type variables' in other words a function who's type is determined by whatever parameters you give it: 

# a generic maximum function of any type T

fun my-max<T>(a :: T, b :: T)-> T:
  if (a > b):
    a
    # if not(a > b) then a == b or a < b   
  else:
    b
  end
where:
  my-max(2, 3) is 3
  # a is unicode 97, b is 98
  my-max("a", "b") is "b"

Big-O explained

The first resource we can use where 'running time' is actually explained is the beginning chapters of Functional Data Structures by Prof Ragde at Waterloo try reading the first few chapters up to 2.2.

The formal definition is:

\(f(x) \le c\cdot g(x)\) for all inputs \(x \ge x_0\) as x (inputs) goes to infinity. If that inequality holds then f(x) is in O(g(x)).

f() function is a cost function derived from a cost model. It's where you count up every constant operation and loop costs by analyzing source code. This is not always easy since what costs do you assign when the code triggers garbage collection or triggers memory allocation because an object has been created deep in some library code.

The g() function is chosen from a list of math functions that represent a 'family of functions'. A family means all functions that are linear, all functions that are quadratic, all functions that are linearithmic etc. It actually is a specific list as we'll soon see.

Why are constants hidden in big-O? If you read SICP their example is 16 and 8 both differ by a factor of 2 and 8 but we don't care which factor. In real life however you will never have infinite inputs so the constant factors do actually matter if they are very large hiding a lot of operations. A classic example of this is when Knuth was writing the Stanford GraphBase (contents are all in Vol 4A of TAOCP) and all his real world testing of graph algorithms found that 'Fibonacci heaps' that were analyzed using a technique called amortized analaysis weren't any better than Kruskal's old algorithm because the analysis was hiding large constants.

O(g(x)) represents an upper bound for all functions in the family g(x). It doesn't represent growth since f(n) = n2 does not grow to be quadratic, f() already is quadratic, thus f() is in \(O(n^{2})\) or upper bounded by every quadratic function. A function n or f(n) is bounded by n, n2 and nn that's why it's an upper bound not the upper bound as there's many upper bounds but you obviously want the closest fitting bound.

Big-O is in a hierarchy of asymptotics there's many more like little-o (strictly less than), big-Omega (strictly greater than), or big-Theta \(\theta(n)\) which describes a tight lower and upper bound. For example a program that searches a single list for a specific element is going to be at least a lower bound of O(n) because it has to look at everything once and upper bounded by O(n) so it is \(\theta(n)\).
