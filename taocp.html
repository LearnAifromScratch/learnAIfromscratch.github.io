The Book(s)

See the official book site, I'm going to start with Volume 1: Fundamental Algorithms. These books aren't a reference, much of the index points to exercises and the content is often comparing two algorithms together. You can decide after you read it yourself how to classify them.

The physical book itself is a representation of the work that has gone into it's writing with high quality typography, perfect paper and a binding where no matter what page you flip to the book stays flat. If you were to ever invest money into a physical book this would be one of those books. A single page can take a week to finish but we'll see how we do. If I were to explain these it would be you learn decades of CS research all at the same time in the kind of detail you will never find elsewhere.

None of these notes will make any sense unless you are reading the book(s) with them I'm not copying the text.

MMIX and MIX

You can if you want use the MMIX Supplement by Ruckert to replace the MIX/MIXAL content with MMIX/MMIXAL in volumes 1-3. MMIX isn't a single machine, you can customize the hardware however you want using Knuth's free meta-simulator program which is a test bed to try out different chip configurations. You can change instruction throughput, pipeline configurations, branch prediction, cache sizes and associativity, bus speeds, anything. He documented that program with the book MMIXware: A RISC Computer for the Third Millennium and all the code is written in literate programming style so you can read the programs like you would any other book. RISC-V is taught in MIT's 6.004 course on YouTube to better understand a RISC architecture if you want. MMIX assumes an operating system exists. MMIX you can use the mixmasters version or the 2017 sealed in time Knuth version you compile from c-web sources that are extensively documented.

To start I'm just going to use the original MIX, and switch in Volume 4 to MMIX but the concepts are the same. We can use the GNU MDK or other software to run MIXware meaning compile .mixal files to .mix and run them either with mixvm, gmixvm GUI, or a scheme interpreter. MIX does not have an operating system so if you like to code bare architecture and do some old school inputs choose that instead of MMIX. Knuth in interviews claims learning MIX is still important because it gives you an idea of where all this modern abstraction decisions came from originally but in some future version of TAOCP he's removing it all for MMIX.

Optimizing compilers are dead

Another reason to learn this material is given by DJ Bernstein's talk a few years ago about the death of optimizing compilers, see these slides where at the end Knuth describes how hand coded assembly will always be faster than the best optimizing compilers. 4390 points to 52, 439c points to 74, if you look in the register state box sp (stack pointer) points to 439a or 50. These are 1 byte each in size or 8 bits. At 44/74 in the tutorial if you move back -0x8 or 8 bytes, count back 8 times or each row of 4 hex values has 2 bytes, so move back 4 rows. If you see 0x2(r15) that means whatever the memory r15 points to, skip 2 bytes. You'll pick it up as we step through the code, and we will step through a lot of code.

Sydney

Set breakpoint at most interesting function which looks like check password. Enter 'test' as text for password, then c to zip to the breakpoint. Pressing s and stepping, r15 is pointing to 439c where we see 7465 7374 or 'test'. Notice the instructions however: 

448a:  bf90 2b5d 0000   cmp 0x5d2b, 0x0(r15)
4492:  bf90 7021 0200   cmp #0x2170, 0x2(r15)
449a:  bf90 3256 0400   cmp #0x5632, 0x4(r15)

It's comparing each byte in the memory that r15 points to some constant hex values by offsetting 2 bytes each time. 1.1 Algorithms

Whenever Knuth cites a reference/paper you can use Sci-Hub to get a copy. Page 7 he describes Tn the running time of just the E1 step in the 1.1E algorith writing that if you were to do this for gigantic inputs eventually you'd discover you could find the value of any n with (constant * ln(n)) or in big-O notation O(ln(n)) where ln() is the natural logarithm function meaning it's the exponent x in ex = y we will shortly learn in the math chapters. We use ex and ln() instead of base 10 log (or even base 2 log sometimes) because they are much simpler to model with than nx since ex derivative is itself and other nice properties.

The 'quadruple' definition of what an algorithm is will make sense when you try it yourself by hand. First we have f, which defines the computational steps:

  • f(n) = n (return the result and terminate, we aren't calling another function)
  • f(m,n) maps to f(m,n,0,1), the zero being just a placeholder to enter to the next step.
  • Step 1: f(m,n,r,1) maps to f(m,n, remainder of m divided by n, 2)
  • Step 2: f(m,n,r,2) maps to f(n) if the remainder is zero, or f(m,n,r,3) otherwise.
  • Step 3: f(m,n,p,3) maps to f(n, p, p, 1) swapping m with n, and setting the remainder p to n. The variable p is introduced here to make it explicit that p does not equal zero, as r can be equal to zero so different notation is used but it's just remainder with a new variable.

Let's try it using Knuth's example of m = 119 and n = 544 since the preface told us to try all these algorithms by hand. 

f(119, 544)  -> f(119, 544, 0 ,1)
f(119, 544, 0, 1) -> f(119, 544, 119, 2) :divide m by n
f(119, 544, 119, 2) -> r is not 0 -> f(119, 544, 119, 3)
f(119, 544, 119, 3) -> f(544, 119, 119, 1) :set m <- n <- r
f(544, 119, 119, 1) -> f(544, 119, 68, 2) :r = 68
f(544, 119, 68, 2) -> r is not 0 -> f(544, 119, 68, 3)
f(544, 119, 68, 3) -> f(119, 68, 68, 1)
f(119, 68, 68, 1) -> f(119, 68, 51, 2) :r = 51
f(119, 68, 51, 2) -> r is not 0 -> f(119, 68, 51, 3)
f(119, 68, 51, 3) -> f(68, 51, 51, 1)
f(68, 51, 51, 1) -> f(68, 51, 17, 2) :r = 17
f(68, 51, 17, 2) -> r is not 0 -> f(68, 51, 17, 3)
f(68, 51, 17, 3) -> f(51, 17, 17, 1)
f(51, 17, 17, 1) -> f(51, 17, 0, 2)
f(51, 17, 0, 2) -> r = 0 -> f(17) 
f(17) -> 17

The last Markov algorithm example is pattern matching on strings. Alpha \(\alpha\) can be the empty string too so what's happening here is we are replacing characters out of strings every loop according to a set of matching rules exampled here. A is a set of finite alphabet characters, and A* is all the possible strings you could make using ordered sequences of characters from A. Next he defines (sigma, j) where sigma is some string in A* and j is a number less than N which is the final state. So input = (string, 0) and output = (string, num) or a termination state.

From this example we have 3 rules, and if the string doesn't match any rule patterns we terminate the algorithm. The set 'I' is the input so f("101", 0). Omega would be the output f("|||||", 8). Both of these are contained in Q and every other intermediary state like f("0|01", 1). There's an exercise for this and I still don't really understand some of the variables he uses but let's move on, I get what an algorithm is.

1.2.1 Mathematical Induction

Proof by induction here is a little different than the usual texts on it in schools such as here. There they assume P(n), and prove the successor P(n+1). Here Knuth proves P(1), then proves every successor input which implies P(n+1) is true each time he does it. The first example, adding 2n + 1 to both sides that's the next successor so if we are at 22 = 1 + 3 then 2n + 1 on both sides is 1 + 3 + 5 = 22 + 4 + 1 or 9 or 32. The bottom of page 14 he then talks about assuming P(k) by induction is true whenever k < n.

The Fibonnaci example, that strike through symbol is the Latin Phi which is a constant here representing 'the golden ratio' because if you divide every Fibn by Fibn-1 you get a better and better approximation to 1.61803.. or Phi. So 3/2 is 1.5, 8/5 is 1.6, 13/8 is 1.624 and it keeps going approaching Phi. This 'divine/golden ratio' is originally from Euclid's book where he paritioned a straight line into segments, then shows the various ratios of each line segment ratio'd against the original are the same and the resulting quotient is Phi. Fibonacci sequence/Phi keeps popping up in astronomy (Kepler) and in plants, chemistry, everywhere. Fibonacci numbers are also a simple model of population growth. In the inequality Knuth factored out the common term from \(\phi^{n-2} + \phi^{n-1}\) to get \(\phi^{n-2}(1 + \phi)\) look up exponent rules if you forgot them (they're in the next section 1.2.2).

The assertions flow chart is similar to programming by contracts where you have preconditions, loop invariants and postconditions. A restriction on the admissible inputs to a function is called a precondition. A postcondition asserts that the output of the algorithm is what you expect or as Dijkstra puts it 'imposing a condition upon the state in which the system must find itself after its activity'. The loop invariant is an assertion that is true before the loop, during every iteration and immediatley after exiting the loop here it is Eqs(6) where (am + bn = d) holds before and after division (it's an exercise later). I didn't know von Neumann invented these. Knuth and that end of chapter Turing quote inform us we should always be adding one or two assertions of the state of the algorithm at 'well chosen points' to actually understand the algorithm, Knuth claims you can't explain the algorithm to someone without them.

First let's understand implication. By elimination rule if assertion A implies assertion B, and assertion A holds as the algorithm goes to the next step (we have a proof of A) then assertion B holds (we have a proof of B). Knuth says A2 implies A6, looking at the A6 assertions gcd(c,d) = gcd(m,n) are implied from A2 assertion c=m and d=n.

1.2.3 Sums and Products

Summation notation is similar to a fold or reduce function in programming. You iterate from start index to n and accumulate producing a final value. The R(j) notation means an inequality relation for the index. Infinite sums are not commutative for example the divergent sum: 1 - 1 + 1 - 1 +… can be 0 or 1 if rearranged: 1 + (-1 + 1) + (-1 + 1).. but it's unlikely we will see many infinite sums until later volumes remember the math prelims here are for every volume up to 5A (then there is a new chapter math prelims redux). You can also come back to this chapter when you come across summation manipulation in practice later in the text.

  • Rule a is simple if you write out some basic examples for yourself though the book example is pretty clear.
  • Rule b says you can rearrange (permutations) the values in the sums all you want using the index just make sure you include everything by adjusting the main sum if needed. The change of variable transformation shifts the sum to start at a new index. The function p(j) for example can be j = k - 1

\[\sum\limits_{k = 2}^{\infty} {k} = \sum\limits_{j = 1}^{\infty} {(j + 1) = 2 + 3 + 4...}\]

The summation index can now start at 1 instead of 2 since p(1) is p(1 = 2 - 1) and nothing in the original sum changed, just instead of starting at k = 2 we now have a transformation p(j) starting at 1. After the indexing transformation add (j + 1) in the sum to keep the same values. We can change the index variables to sum backwards if we wanted or any permutation.

  • Rule c the double index aij is a matrix or in the examples a 2D array. It says we can commute the two sums in a product of sums and he shows how they end up being sum + sum so commutable anyway.
  • Rule d sum 1 to m, then sum m to n, so we are summing m index twice hence the domain now being 1 to n + am
