Intro

I did the following:

  • read papers from compsci professors who specialize in research based teaching
  • collected as many anecdotes as I could how others learned because sometimes anecdotes are worth doing
  • asked everyone I work with, who have very fancy degrees (I don't), that if they were to design a modern compsci curriculum what would they think is essential to know

This is the result. It's actually a 'how do I get paid to do this' workshop but we will go deep into some computer science topics this isn't a bootcamp. When we switch to graduate material I won't tell you so you don't psych yourself out into defeat thinking it's too 'hard' it won't be any harder than when you first started.

Curriculum

I made everything a task/project since all of this is learn by doing. You can decide exactly how much of the attached course materials you need to research to finish the tasks. I'm doing all the tasks, you do what you want to do.

  • The USACO guide from scratch as we finish the following including advanced data structures and designing of algorithms
  • A workbook on languages
  • Task of some reading on software engineering (code review, design specs)
  • Project to build an app to beta release (any language)
  • Tasks to add features to a kernel and build network device drivers (C or any lang that compiles to C)
  • Tasks to build MapReduce, Faul-tolerant key/value storage and Part 2, Sharded key/value service (Go)
  • Project to implement something fault tolerant to put on a public GitHub (any language)
  • Tasks to write features for Bril (TypeScript, OCaml or Rust is easiest)
  • Tasks to complete the features for BusTub relational DBMS (C++)
  • Task to write a foreign data wrapper for PostgreSQL (any language)
  • Project to complete a design document and implement a major OLAP system feature for a personal GitHub (any language)

You are a real developer now, not a 'junior' developer never give yourself underwhelming titles.

Task 1

You won't have to download any software and can do this on a phone if needed.

Choose one of these languages. Whatever language you choose we'll only be using a small subset of it's standard library so it really doesn't matter what you choose. You probably already have a language in mind you want to learn so chase what you want. Here is some recommendations, use library genesis to get a pdf if you can't buy/borrow them:

  • The Brown University Rust Book experiment. They rewrote the Rust book to improve understanding of the borrow checker and other Rust features
  • The OCaml book from Cornell's CS3110 and Real World OCaml
  • The GNU C++ Manual though we will learn learn C++ from USACO as well
  • Kernighan's Go book
  • Haskell course Programming with Categories (MIT) it assumes you don't know Haskell and has lectures
  • Richard Stallman's GNU C Language Intro and Reference Manual, or the ancient Kernighan & Ritchie book, or Prof Ragde's (uWaterloo) If You Must Learn C. The Stallman intro is quite well written and is new as of Sep 2022.
  • ANSI Common Lisp or Gerbil Scheme
  • The Swift book by Apple

Remember we are only using enough of the language's standard library to understand algorithm design. Our goal is to repeatedly see problems, apply some algorithm design strategy, write trash code to solve the problem. By doing this over and over we will eventually gain the algorithmic mindset and this will all become second nature to us. We will learn 'safe' code or non-trash code later doing projects for now just solve on cruise control by any means necessary using your preferred language.

Kattis accepts APL, Prolog, Fortran and many 'what the hell is this' tier languages. Stallman once wrote an entire editor in APL somehow.

How to Program

Read the book of the language you chose or use the USACO guide first chapter to teach you how input/output works in C++/Java/Python. There's the USACO online IDE but 90% of problems I only used the Kattis editor as you have unlimited submissions.

List all the problems by easiest and start with the very first one 'Velkomin!'. Many of these early Reykjavik university problems have you covering everything needed they're very good. You solve these by writing examples for yourself of input output for corner cases.

The problem Quality-Adjusted Life-Year is an example of how to take multiple lines of input. Many of these problems the first input will be how much more input to expect which you can then loop on: 

#include <iostream>
using namespace std;
int main(){
    int tt;  
    double sum = 0;  // declare a type double and set it's value to 0 
    cin >> tt;   // standard input pipe to our int tt
    while (tt--) {  // is false when input empty 
        double a, b;  // each loop declare these 2 new assignables wiping out the old ones (kindof, we'll learn this)
        cin >> a >> b; // take 2 inputs
        sum = sum + (a * b);  // accumulator
        }
    cout << sum;
}

Notice I didn't even include a 'Return 0' in main() we can worry about safe/proper code later for now just get familiar with programming and see how far you get. The hardest part right now is dealing with input/output so I will delve as far as I can into C++ I/O.
