What is this

How to game the current technical interview system for fun and profit. We need to be able to do this: "it's not enough to be able to solve the problems, you need to be able to solve them on the spot, without much thinking, in an interview context". Codility a popular filtering site for testing applicants based it's service on the IOI competition, where all problems usually have an easy but 'suboptimal' solution, and the optimal solution requires algorithmic skill, so let's get that skill.

I assume you've seen some of this before like CS19 and math proofs but I try and explain everything but it would help if you've seen all this before at least once, if not do this twice you'll get more out of it, this is how I learned by repeatedly doing the same material and each time finding a new insight I missed because you kind of panic the first iteration just trying to keep up. When you go back with experience you keep finding new things they were trying to teach you.

How this works

I collected the most efficient and interesting resources that are just above our skill level so we aren't bored but it's not impossible either, and we learn actual useful things. I removed things that are taught in undergrad but never used in grad instruction or in industry. I try to keep lecture watching time to a minimum and the one's we do watch will be worth it and something you couldn't just read yourself or watching it is faster than reading.

I do the materials with you and anything that seems like they assume a background I work it out in the notes otherwise you don't need these notes and can quickly skim them. I also try sometimes to explain something needlessly complex in the most plain language possible, try this yourself it's a great way to learn by having to explain it all.

OCaml crash course

Here's a quick rundown how abstract OCaml can get. 

# let compose f g = fun x -> f(g x);;
val compose : ('a -> 'b) -> ('c -> 'a) -> 'c -> 'b = <fun>

# compose (fun x -> x + x) (fun y -> y + y) 2;;
- : int = 8

Above is a function composition f(g(input)) where a function g is applied first to an input and the result is the input for function f. Lambda syntax (a function without a name) in OCaml is fun x -> x and you can bind that anonymous function to a name with let. 

val compose : ('a -> 'b) -> ('c -> 'a) -> 'c -> 'b = <fun>

These arrows form typ -> typ' where typ is the domain type (the type of the inputs) and typ' is its range type (the type of the result). The last -> 'b on the right is the result, and everything else to the left denotes the parameters that function takes. Arrows with brackets around them (typ -> typ') denote a function is expected as a parameter. These are all type variables meaning they change to whatever data you give it, if you give it strings it's now type string. The type variables 'a, 'b and 'c are alpha, beta and gamma. Think of them like basic math variables: a + b = 2 where a and b can be the same value 1 or different like -2 and 4, it's the same deal with type variables 'a -> 'b can be int -> int or int -> string but should be consistent or you will get a type error. The = <fun> notation is a placeholder for the byte code generated by the top-level since it would make no sense to display the byte code to you in the top-level. 

let g x =
  let f = fun y -> y + x in
   let x = 1 in
    f x

OCaml developers often use nested evaluation sometimes called inlining which is a way to both limit scope like importing a library who's namespace can't escape the let binding and for compiler optimization as you give it all it needs at once like Common Table Expressions in SQL 'here's everything I want to do as a snippet, you figure out how best to do it'. At first you will hate this style then wonder why every language doesn't have this style after a month of using it. 

# let g x =
 let f = fun y -> y + x in
  let x = 3 in
   f x
val g : int -> int = <fun>

Eval is by substitution:

# g 10
-->
let f = fun y -> y + 10 in
 let x = 3 in
  f x
-->
let x = 3 in
(fun y -> y + 10) x
-->
(fun y -> y + 10) 3
-->
(3 + 10)
-->
13

It's just the substitution model, the same thing from highschool math. The step (fun y -> y + 10) 3 is similar to the lambda calculus syntax: function on the left applied to value on the right and evaluated via substitution until you reach a value. In ML-family languages functions are a value. 

# let rec power f n = if n <= 0 then (fun x -> x) else compose f (power f (n-1));;
val power : ('a -> 'a) -> int -> 'a -> 'a = <fun>

# let sqr = power (fun x -> x * x) 2;;
val sqr : int -> int = <fun>

# sqr 2;;
- : int = 16

A power function that takes another function, and the number of times you want to square that function. The rec keyword is to get around the problem of a function that references itself before it is defined, otherwise the language will tell you power is an unknown function when it first reads the body definition.

The nested parens I probably screwed up and variables are confusing but remember OCaml applies (function) to (value) or f v. Try hand stepping it yourself, recursion is delayed computation you keep looping until you hit the base case which here is 0 and returns x -> x the identity function, note it is not returning n = 0 the counter that power uses. 

# let derivative dx f = function x -> (f(x +. dx) -. f(x)) /. dx;;
val derivative : float -> (float -> float) -> float -> float = <fun>

The dot notation +. or /. denotes a float is expected. The type arrow notation denotes a function that takes a float and a function as input, and returns a new function from float -> float.

This calculates the derivative of an arbitrary function f. Imagine a power function f(x) = x2 and you want it's derivative, which uses the formula f(x)' = nxn-1 where you bring down the exponent, subtract 1 from the original exponent, and multiply or f(x) = 2x1 with the derivative dx f above function being the longer version of that to calculate any derivative not just a power function. 

# let squared = derivative 1e-5 (fun x -> x *. x);;
val squared : float -> float = <fun>
# squared 4.;;
- : float = 8.00000999952033

# let three = derivative 1e-5 (fun x -> x *. x *. x);;
val three : float -> float = <fun>
# three 2.;;
- : float = 12.0000600002612128

I passed derivative a tiny dx, and a power function which is f x = x2, it returned the derivative of f x = x2 which in calculus is another function f x = 2x. I applied that to 4 and got 8. The three derivative is f x = x3 or using the power derivative formula it's f x = 3x2 and I applied that to 2 and got 12 as expected.

Now we can go crazy: 

# let pi = 4.0 *. atan 1.0;;
val pi : float = 3.14159265358979312

# let sin''' = (power (derivative 1e-5) 3) sin;;
val sin''' : float -> float = <fun>

# sin''' pi;;
- : float = 0.999998972517346263

'sin' and atan are both built-in OCaml functions. What happened here was I created pi, a constant value. Then I calculated the third derivative of sine, meaning take it's derivative 3x, by computing the cubic power of the derivative function, which returns a new function, and applying that to sin. I could then apply that final returned function to pi and get back a number.

So a partially defined function is returned then partially defined again and then applied all without having to write new functions. Completely abstract style of programming. 

# let sin''' = (power (derivative 1e-5) 3) sin in
   sin''' pi;;
- : float = 0.999998972517346263

And finally as a common idiom in OCaml combining everything into one block which reads: let variable sin''' be the function that is returned from (power (derivative 1e-5) 3), apply that returned function to sin (built-in) and then apply whatever is returned to pi.

This calculus example was lifted directly from the book Using, Understanding, and Unraveling the OCaml Language and exists here to explain the example if you choose to read that paper (sci-hub has a better version than the above link) and showcase the kind of crazy abstract passing around of functions and partial application you can do in OCaml. It's module system is like this too you can break up programs into independent pieces that can all be compiled seperately and then change these modules as needed when you use them on the fly.

Tools and techniques

The first thing we should do is read some of Tools and Techniques chapter from Functional Data Structures to explain the math notation of basic algorithm analysis. Prof Ragde researches algorithm design/analysis and discovered that first year students could not 'think algorithmically or write code from scratch' so he developed a functional language approach. Prof K from CS19, the other workshop here does the exact same thing and they both frequently collaborate to fix education, so if you've taken CS19 some of this will be familiar to you.

We will redo all this in a lecture this is a first read.

2.1.1 Ambiguities in mathematics uses the Prof K CS19 version of Big-O which is lambda/anonymous function O([n -> n2]) to clarify what O(n2) really means: input size n and the growth is n2

2.1.2 A formal definition we will shortly learn that f(n) consumes input sizes and returns running time, or complexity, or whatever you want to call it. Using a simple model you add up costs of running your program, which is always a function of the size of the input. If your program does a single loop on the input each iteration doing 5 sequential things then f(n) = 5n. The g(n) in O(g(n)) is an arbitrary function you choose from a list of known math functions that best describes the growth of f(n). We will learn how to choose these functions. The statement \(n \ge n_0\) means every input n that is greater or equal to some input threshold n0 but as Prof Ragde notes nobody in algorithm analysis actually uses the constant or n0 to make a precise bound since the whole point of big-O is to allow for wiggle room in the definition to ignore constants so we can make a worse case one-analysis-fits-all for every kind of computer architecture. Giving a single toss about the precise constant and input threshold is a total waste of your time when working in asymptotics even though many tutorials, courses and textbooks focus on these things.

2.1.3 The sum theorem means O(n + n) is just O(n) and O(n2 + n) is O(n2), you choose the max bound of all functions. The small stuff theorem if you ratio f(n) over g(n) and the inputs near infinity, if the limit is 0 then f(n) is in g(n). For example n/n2 approaches 0 when given gigantic inputs and n/n is 1 or a constant so n is in n. An example of using the constant O(1) notation is a polynomial n3 or n6 if you want to abstract away the specific value use nO(1) meaning the exponent is a constant.

2.1.4 His comment about C++/Java is because of resizeable objects being placed on the heap (memory in RAM, as opposed to stack memory allocation that is closer to CPU) and because of garbage collection going on in the background and other language specific things add extra hidden costs. You can learn this specific model from Knuth's PhD student Robert Sedgewick who has many algorithm texts for Java/C++ if you want. Another model that CMU teaches is a language based model taught in their parallel computing book that uses a language similar to SML/OCaml. If you browse the public courses here on 'Diderot' and click on all the semesters of 15-210 you'll find recorded TA lectures/recitation and practice sessions, and the entire book they use. Rant: every university changes their ed-tech seemingly every 2 years, so Diderot won't last and they will use some other awful ed-tech abomination soon so another way to get this book is from library genesis searching for Parallel and Sequential Algorithms. It truly is a fantastic book, showing you how to break up the work/span of common sequential algorithms but unfortunately no time to do it here but I highly recommend it. Search github for the 15-210 labs, then write your own solution from scratch and compare it to other students as a solution guide.

2.1.5 'word size' is the largest maximum addressable memory your machine can address, which is probably 64bit. He makes a note about polylogs, which we will again see shortly, which are log n factors that creep into analysis with the comment that you want to keep things vague and when you try and make things precise in algorithm analysis you run into problems though we'll see an asymptotic called O-tilde used to ignore these extra logarithmic factors.

2.1.6 Algorithm analysis in practice the Sequential and Product theorems are all you will ever need to read imperative pseudocode (the code given in most algorithms books), assign costs, and figure out the big-O worse case analysis. Any sequential program that finishes a computation and moves on to a new one you add up all the costs to functions that are dependent on each other, assign them big-O values, then choose the max of all these functions and that's your overall program complexity. A loop you multiply whatever is in the body by the input, which typically is n * constants but if another loop of n size is called it's n * n sometimes called nested loops, but don't overthink whatever is there multiply it by n unless you are throwing away input each loop. That's it.

The end of 2.1.6 he tells us that there is so much going on with running a modern program that algorithm analysis is only useful for getting early information in the design process so we are learning all this just to write a design doc and likely throw out most of our design as reality factors in and we have to change everything. This is something else most algorithms courses and texts never tell you.

Let's figure out his \(\widetilde{O}\)(g(x)) definition. He took \(n^5 \cdot 3^{n}\) who's form is g(x) * poly(log(n)) or n5 * poly(log(3n)) and log 3n is n so this means n are the polylog factors you can suppress, the n5 factors. Harmonic numbers

Watching part b. Harmonic numbers will come up in every algorithms text. This is a nice demonstration how you would use asymptotics to estimate an upper bound/lower bound of a sum. @9:44 an integration trick to find the upper bound of harmonic numbers. Integration is an undo function for derivatives to turn a derivative back to the original function so ln(x) it's derivative is 1/x, reverse that then this range is 1 to n or a to b and the integral eq is f(b) - f(a) where you plug in those range values. ln(n) with n plugged in is just ln(n), ln(1) is 0, so it's ln(n) - 0.

Did that seem really easy to you instead of the very confusing Riemann sum approximation and silly limit of infinite rectangles they teach in every calculus course where they go on about the area under a curve it's because I used the classic Newton calculus integral which we learn in the math workshop that any elementary school kid could learn.

Part c is worth watching if you're interested in more asymptotic tricks, at the end he minimizes functions with parameters to showcase how weird bounds end up in papers that you wonder how on earth did they find this obscure bound.

Factorials

Last lecture Factorial Asymptotics. This will come up in the first chapter of the competitive programming book. For the lower bound of n! put in some examples: 2! = n(n-1) which is (22 -2) or 2. 3! = n(n-1)(n-2) which is (32 -3) and 6(3-2) or 6. Is 2! at least 2/2 or 1, and is 3! at least 3/2 or 1.5? Sure, so lower bound is n/2 multiplied n/2 times as he cut off half the lower terms to make this an easy equation. By now you should be seeing that the whole point of asymptotics is to estimate and make everything easy so you can work at a high level scratchwork tier and not have to crunch any complex equations.

His advice is take logs of these big products and expressions which is exactly the reasons logarithms were used in the 1600s to turn large products into easier sums. Recall the log laws: ln(ab) is b(ln(a)) and ln(a/b) is ln(a) - ln(b). @4:08 this is 1/2(n(ln(n) - ln(2))), distribute n: 1/2(n(ln(n)) - n(ln(2)) and of course this is - O(n) as we suppress ln(c) constant if you're wondering where the negative O(n) comes from.

He begins using an example of hash table (array) doubling. We already did hashing in CS19 where the idea is to compute efficiently on the keys in a balanced binary search tree and each key maps to an array index. The array (table) is size m and the number of keys you insert is n. When the array fills up you want to increase the table size, so you double the size of m and copy over all the old array data and add the new insertion n. Then when double m is full double again. When he says 'you only double log n times' and proceeds to demonstrate a growing geometric series of doubling recall that 2log n is n. Suppose m = 1. When you have 2 keys you double to 21, 3 keys you double to 22, 5 keys you double to 23. If you stop at 8 keys, you have doubled 3 times, or log(8) = 3 or log(n) times.

He explains 2-3 trees later but we don't need to know what a 2-3 tree is to understand the example just the interface that insertions/deletions cost O(log n). When he was adding up the total costs he ignored constants and reasoned deletion was at most 2x the inserts so 2i(log(n)) which is O(i(log(n)). Deletions were free here because the total cost ended up being O(c + i(log(n))) and when you divide up that total over all operations deletions end up 'costing nothing' in amortized costs.

@41:40 this is the method we're interested in, the potential method. Note that we are using it per operation, so amortizing the cost of incrementing +1 operation or inserting a new node operation. After enough examples of 2-3 trees and 5-2 trees we see that anything that ends up in a 'bad state' where the next operation generates a cost such as splitting or merging is what you use as your potential function phi(end state) or phi(after op). I guess the intuition here is that the potential function builds up cost energy with every operation, and the goal is to drain that cost by subtracting the expensive end state from the beginning state of an operation and hopefully getting a constant.

The formula: actual cost of op + phi(after op) - phi(before op)

In Tarjan's original paper, he uses an example of a stack operation, defining the potential function of a stack to be the number of items it contains, which is of course non-negative as required. If we have a stack with n amount of items, and we perform a sequence of operations that does k pops (removals) and 1 push, then we have

  • actual cost: O(k) + O(1)
  • before op: n (amount of stack items already there)
  • after op: n - k + 1 (the new total n)
  • Total + after - before:
    • (k + 1) + (n - k + 1) - n = 2.

The k pops cancel out, n cancels out, amortized cost of that sequence is constant. This is something we have to see by example over and over to really get it which we will in 6.851
