Input/Output

Read the book or online documemtation of the language you chose or use the USACO guide first chapter to teach yourself how input/output works in C++/Java/Python. There's the USACO online IDE too if you want.

The problem qaly is an example of how to take multiple lines of input. Many of these problems the first input will be how much more input to expect which you can then loop on: 

#include <iostream>
using namespace std;
int main(){
    int tt;  
    double sum = 0;  
    cin >> tt;   // pipe first stdin to tt
    while (tt--) {  // false when input empty 
        double a, b;  // each loop declare these 2 new assignables
        cin >> a >> b; // take 2 more inputs
        sum = sum + (a * b);  // accumulator
        }
    cout << sum;  // pipe sum to stdout
}

Finish up to 1.4 Easy problems on Kattis or until you get how I/O works in your language.

Big-O explained

The first resource we can use where 'running time' is actually explained is the beginning chapters of Functional Data Structures by Prof Ragde at Waterloo try reading the first few chapters up to 2.2.

The formal definition is:

\(f(x) \le c\cdot g(x)\) for all inputs \(x \ge x_0\) as x goes to infinity. If that inequality holds then f(x) is in O(g(x)).

Full explanation:

  • x floating around by itself or f(x) = x or y = x represent the same thing.
  • the big-O definition doesn't care if c = 1 or c = 1 trillion.
  • if inputs 'go to infinity' traditionally you write n instead of x.
  • x0 means some specific input of x or threshold.
    • can be the first input or the trillionth input the big-O definition doesn't care.
    • all inputs are positive as it would make no sense to model negative input.
  • f(x) is a cost function.
  • g(x) is chosen from a list of existing math functions, see below.

Toy example: Suppose a program consists of 2 constant operations followed by a single loop with 10 constant operations in the loop body. In the RAM model this is f(x) = 10x + 2

  • f(n) = 10n + 2
    • n because f(x) 'goes to infinity'
  • 10n + 2 <= cn
    • true if c = 11 for all inputs x >= 2
    • also true if c = 12, a thousand, a billion

f() function is a cost function derived from a cost model. It's where you count up every constant operation and loop costs by analyzing source code. This is not always easy since what costs do you assign when the code triggers garbage collection or something triggers memory allocation because an object has been created deep in some library code or for functions that can consume other functions as inputs.

The g() function is chosen from a list of math functions that represent a 'family of functions'. A family means all functions that are linear, all functions that are quadratic, all functions that are linearithmic etc. It actually is a specific list as we'll soon see.

Why are constants hidden in big-O? If you read SICP their example is 16 and 8 both differ by a factor of 2 and 8 but we don't care which factor. In real life however you will never have infinite inputs so the constant factors do actually matter if they are very large hiding a lot of operations.

O(g(x)) represents an upper bound for all functions in the family g(x). It doesn't represent growth since f(n) = n2 does not grow to be quadratic, f() already is quadratic, thus f() is in \(O(n^{2})\) or upper bounded by every quadratic function. The function n is bounded by n, n2 and nn that's why it's an upper bound not the upper bound as there's many upper bounds but you obviously want the closest fitting bound.

Big-O is in a hierarchy of asymptotics there's many more like little-o (strictly less than), big-Omega (strictly greater than), or big-Theta which describes a lower bound and an upper bound. For example an algorithm that searches a list is going to be at least a lower bound of O(n) because it has to look at everything once and upper bounded by O(n) so it is \(\theta(n)\).
