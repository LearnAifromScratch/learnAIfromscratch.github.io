Being a wagie

To be a software developer, not a 'junior' or 'intermediate' or other fake title you should never give yourself, you simply have to prove to someone that you can teach yourself as needed, that you know the basics of problem solving strategies (divide and conquer, dynamic programming etc), that you can reason about the basic efficiency of your code (basic big-O and amortized analysis) and that you have personal standards for your work meaning you care that it is safe code and doesn't look like a pile of garbage. The most critical factor that I've seen sitting in on interviews is the ability to teach yourself as needed because there is no possible way to prepare for every kind of wagie software job and all their internal tooling they use.

The kinds of wagie entry slots we are targeting are not advertised by companies because they end up getting a hundred resumes you instead target a place you've always wanted to work at and get your name to whatever hiring manager either on LinkedIn or general application.

We don't have anyone to vouch for us who is presently working and worked with us before and we don't have any official piece of paper touting our credentials. This means we should expect filtering tests by outfits such as codility or hackerrank. Read this paper Codility. Application of Olympiad-Style Code Assessment to Pre-Hire Screening of Programmers. Here is some of the content:

  • The interviewers observed that an easily identifiable reason of many failed interviews was the inability to write bug-free code
  • There usually exists a trivial but suboptimal solution and the ability to produce such a solution proves only elementary programming skills while producing the optimal solution requires algorithmic skill.

Both these problems are solved by doing basic competitive programming resources. Bugs in interview style code, for the entry level jobs we are aiming at, is a case of misunderstanding the problem and not having enough experience. Similarily an optimal solution can only be found if you've seen it already implemented. Algorithmic skills they don't have time to teach but everything else like the business methods they use to write software and custom tools can all be learned easily as a wagie and everyone does it differently. For example all other curriculums will insist you learn git while many companies use a mercurial fork they've customized or cloned like Google's custom Piper, Facebook's Eden, Mozilla or Jane Street who wraps commands around hg. The reason is you can architect Mercurial beyond a file system model to instead make it into a distributed database which we will learn here. The point is you would never be able to teach yourself these internal tools they are learned on the job which is why the ability to teach yourself and figure it out on your own is key to being a wagie.

tl;dr clearly we should take some competition programming training which we will in the form of the USACO guide.

Being a freelancer

You can start doing this walking around your city, or even dealing with the sometimes crazy people who offer craigslist work. Local is always the best way because you are a real person they can physically see and know how to get in contact with. You have to build a network like meeting somebody who has a lot of airbnb businesses, or a restaurant, or some vintage clothing store with no online presence. You can also do this being a wagie and leaving with a network of people you know who will pay you. You can also do this if you have an impressive pedigree/resume and tell companies to pay you as a freelancer.

I have used in the past existing online outfits you probably know about and none of them are worth working for post 2020 unless you are willing to put in many years building a reputation and consistently doubling your fees until it's worth your time.

You always charge daily not hourly, you always increase your rates with new clients. If someone accepts your rates without question then double it for the next client. You bill in deliverables which you negotiate clearly like if somebody wants a website made then one deliverable is a specific feature, adding authentication is another deliverable etc. You want to always be delivering something because without fail the requirements will change from the client. Never disappear for a month or two then produce work make sure you are delivering at least weekly.

I dabbled in many bounties like gitcoin or gitpay but always some guy claims it immediately, spends 2 months producing nothing, they accept my patch which was ready in 2 weeks but then I have to argue with the original guy to get paid who still produced nothing (though once a guy cut and paste my code, merely changed variable names and claimed it as his own then demanded he should be paid first). Someone needs to redo bounties so they are effective (startup idea for you?). There's a lot of this going on especially if you look at ycombinator's 'work at a startup' site anyone who has a public git will be offering money for patches. Pretty sure geohot's tinygrad repository offers them too.

Business of software

If you want to start a software company with the intent of selling it then do not create a LLC. Limited Liability Corps are for lawyers and contractors or side projects. You want something that can issue shares like a S corp in the US which can be turned into a C corp later if needed. No venture capitalists will invest in an LLC but you can convert one to another which looks like a huge hassle so may as well just start with a regular corporation if the intent is being sold or raising capital. If the intent is running a small side business then just get a sole proprietorship so you can get a business bank account. You're not going to be sued if you're not involved in industries like finance/crypto or something regulatory like health or whatever and most of the LLC protections touted aren't that good anyway. The best advice for a startup is here where your goal is always to capture a majority of the market in some form of monopoly avoiding competition but not everything needs to be a billion dollar valuation we can sneak in here and see what happens.

There's a lot of bad software with no real competitors and when we start researching we realize they either solve the problem of insurance/compliance with some government interference in the industry or provide direct customer support meaning pick up a phone and say fix this for me and it happens.

A compliance and insurance example is weed store software in the US it is all pure garbage looking like a kid wrote it and charging enormous monthly fees. That's because Metrc has a monopoly on compliance and if you want to write software that is compliant you have to physically take courses in every state you want to operate in. You also have to guarantee very large fines for any mistakes in reporting and of course nobody can do this unless they have millions of startup capital so a monopoly exists of Dutchie and similar software which is just another tax on whoever runs a dispensary. Stock market software is the same you must send real time reports to regulators and failure to do so is an enormous fine but there's much more competition in this space due to the giga market it serves so you get much better software at least.

Another insurance example is Google/Microsoft. Many apps have to pay security contractors like Bishop Fox for a compliance audit in order to offer google integration. Microsoft is typically the only software you'll find in most main street style businesses because of the insurance risks, using other software requires significant cost into compliance and assessment regarding the privacy of user data or financial transactions. Instead you can pay the Microsoft yearly licenses and be done with it.

That eliminates some software we can build ourselves however online analytical processing (OLAP) we can build because it's always running analytics on public information. We will learn how to build these here and even how to work for the tons of startups that are out there for OLAP data or maybe you can make your own using techology nobody else is.

Task 1 Kattis

In competitive programming you submit source code as a solution to a problem and the online judge compiles and runs your code against tests. There is a huge list of online judges available like Kattis, SPOJ, Timus, Aizu, and more worldwide. This is how we practice to get skilled in algorithm design and problem solving.

Begin task!

Choose one of these languages. List all the problems by easiest and start with the very first one 'Velkomin!'. Many of these early Reykjavik university problems have you covering everything needed for I/O they're very good. Kattis accepts APL, Prolog, Fortran and many other wtf tier languages so be sure to pick the language you like the most not C++ we'll learn it later in the USACO guide. Stallman once wrote an entire editor in APL somehow.

Input/Output

Read the book or online documemtation of the language you chose or use the USACO guide first chapter to teach yourself how input/output works in C++/Java/Python. There's the USACO online IDE too if you want but Kattis accepts multiple submissions so you don't even need an IDE or text editor to start.

The problem qaly is an example of how to take multiple lines of input. Many of these problems the first input will be how much more input to expect which you can then loop on: 

#include <iostream>
using namespace std;
int main(){
    int tt;  
    double sum = 0;  
    cin >> tt;   // pipe first stdin to tt
    while (tt--) {  // false when input empty 
        double a, b;  // each loop declare these 2 new assignables wiping out the old ones (kindof, we'll learn this)
        cin >> a >> b; // take 2 more inputs
        sum = sum + (a * b);  // accumulator
        }
    cout << sum;  // pipe sum to stdout
}

Finish up to 1.4 Easy problems on Kattis or until you get how I/O works in your language.

std::cin is synchronized to std::cout and performs default buffer flushing operations so to save some runtime he disables this sync. His compiler flags he explains in one of these Twitch streams, or you can figure out yourself too as you go, most of them are to make compiler messages extra verbose for debugging. Time complexity

Skimming through time complexity. This is a nice writeup how big-O works and how USACO problems typically are under 100 in input size so we can get away with order of n3 to n4 complexity or 3-4 nested loops. What you should remember here is that everything in a loop is multiplied by how many times it's being iterated so if you have 100 constant ops per loop and your loop is size n then it's 100n or O(n) since constants are suppressed in the definition. If you have another loop nested going through the entire input twice then it's n * n. Complete search or brute force where you try every possible solution is exponential and possible here if inputs under 20.

The 4 suggested resources:

First is 'IUSACO' 3-Algorithm Analysis here. It's only chapter 3 and 4 pages long. The grading server can do 100m operations per second. We will read a paper soon on how wall clock time vs CPU time works giving us a crash course on how to analyze algorithms on real life hardware. Page 8 where did the 5n + 17 come from? If you had a program that had 17 constant operations then performed a loop n times with 5 lines of constant ops in the body, and if you were using a model of computation like the ram model meaning memory can be accessed and written in O(1) constant time then the function to describe the complexity of your program is f(n) = 5n + 17 or O(n).

Next is 'CPH' 2-Time Complexity here from the competitive programming handbook. This is already known to us from the intro courses so let's look at 2.4 Maximum subarray sum. If it doesn't make sense it means which sequence or combination of neighboring elements produces the greatest sum. Try to solve this yourself before reading their solution. Remember you can prep something with multiple linear consecutive runs since O(n + n) is O(2n) which is of course O(n) and in this problem keep removing the negative first and last array element since why would you keep them. The solution they give is Kadane's Algorithm.

Next is 'PAPS' 5-Time Complexity here and the wall clock (using time() in the shell) is again mentioned but we'll learn you can't rely on wall clock time. Insertion sort we already did in the intro to CS courses where we learned for small data insert can be a constant operation. Page 91 he substituted ti for i and since this is a sum where the counter starts at 0, it's going to sum 0 + 1 + 2… up to n - 1 times and we already know a sum from 1 to n closed form is (n + 1)n/2 or here (n - 1)n/2 but we can already see the dominating power is going to be n2 and this is followed by an example how n2 is both an upper bound of n2 and n. The proof of the bound first he noted that if you multiply N >= 1 you get N2 >= N and this fact is used in the inequality to help reduce that side to 2N2 a trick we learn in The Mechanics of Proof.

The video they suggest goes through how to find function f(x) by analyzing the source. @11:03 remember you multiply everything in the body of a loop by the amount of times you loop. Without doing any of the analysis I can immediately tell this is n * c * n3 or n4

Caveats

  • not all loops perform an expensive operation during every loop so we will learn amortized analysis
  • a language based model of computation learned in Functional CS is the modern way instead of pointer machine or ram models that assumes everything is constant read/write
  • in virtually all papers I've read in CS they try and nail down constant factors by showing a multiplicative error or by taking it's log show an additive error and you often wonder where did they get that bound, if interested then take the advanced time complexity material below.

C++ STL

C++ has versions C++98/03/11/14/17/20/23 these all represent years there was a new standard. The current state of C++20/23 or 'modern' C++ is you can do functional programming with it. Options, lambdas, higher-order functions, monads and more all now exist in the definition depending which compiler you use. Google has rage quit C++ and is writing their own replacement called Carbon and as a result clang/llvm has not even implemented all C++20 features. If you want to use C++20/23 in some project you have to use GNU gcc or Microsoft msvc.

Competitive programming typically only accepts C++11/14/17 which we can refer to online documentation for thus we really only need to learn the STL or the standard template library. The STL to me seems like a big hack to make C++ a generic programming language a concept invented by ML in 1973. If you took the Standard ML course listed in the intro to this workship then generic programming is simply polymorphism or 'a types. If you took the accelerated intro course then generic programming is the same as 'type variables' in other words a function who's type is determined by whatever parameters you give it: 

# a generic maximum function of any type T

fun my-max<T>(a :: T, b :: T)-> T:
  if (a > b):
    a
    # if not(a > b) then a == b or a < b   
  else:
    b
  end
where:
  my-max(2, 3) is 3
  # a is unicode 97, b is 98
  my-max("a", "b") is "b"
end

The C++ STL was written by Alexander Stepanov and if you look through the source of the STL you see a bunch of template<typename x, typename n,> type code. These are templates to enable generic types in C++ functions. He wrote a book about it and we can audit the entire book right now because it's only 200 pages or so. Go to a library genesis domain you have access to and get a copy of Elements of Programming.
