Start here

I assume you are teaching yourself at the same time some kind of intro course you found on YouTube or a book somebody recommended such as:

  • Accelerated Intro to CS
  • Functional CS
  • SICP
  • anything

This exists to solve two problems.

The first is finding a job or freelancing which seems like it's next to impossible these days but they are desperate for developers who know the skills we'll be doing here.

The second problem is recruiters always ask me where to find candidates for fintech roles like 'quant developer'. I pressed them on what exactly they look for in 2024 then found free resources that taught as close to the requirements as possible. Everything here was vetted by them as being desirable as a basic foundation (USACO, plai.org, queueing theory, some experience with low-latency network kernel bypass) then the character of the candidate is the determining factor not their claimed experience or credentials. What they want is someone with the ability to figure out things themselves but who knows what they don't know meaning they won't bluff their way through meetings or sabotage the code with unexpected side effects. This person can write software for high throughput applications and can describe in detail the language they are using meaning how the memory abstraction works and if it's a managed runtime how to tune or avoid triggering unnecessary garbage collection. If you possess the above then they claim you can be taught everything else.

Campus recruiting used to cover this but today is monopolized by big tech corps and government crony-run contractors that manipulate school guidelines by pressuring students with 2-week expiring offers and paying the school to lock out all competition so it's a struggle to find anyone qualified to do this kind of work as many companies can't even break into the campus internship or graduate hiring pool.

Strategy

Find a mentor. There is someone somewhere doing what you want to do and making a lot of money doing it and your task is to complete enough material here to reach a level where you can get in with them somehow and absorb everything. Once you are in all you have to do is be the reliable person they turn to whenever some (tedious at first) work needs to be done and if this work over time becomes increasingly more skilled then you have found the right mentor and you are set for life. If they get promoted you get promoted. If they go off and start a new company you are the first hire. If they get hired at a top trading firm everyone says you have no chance of getting into guess who they will tell the hiring managers they should also hire.

You approach this person and sell yourself telling them you are available if they have some work because you want to learn it all from the ground up. There is always something to do and your goal is to be this magic portal where tedious work they despise doing can be dumped and it returns as per specifications they gave you. If you can do this then there's time for them to show you things. You eventually make your way up to highly sophisticated work until you end up being essential to them where they couldn't imagine going back to working without you because it's a mutually beneficial relationship. I actually took all the above advice from a comedian many years ago and it worked. In fintech, this can be tricky because of signing multiple NDAs even between internal departments but if you sell yourself to them that you are serious about being mentored they will tell management to make it happen.

Input/Output

Read the USACO guide first chapter to teach yourself how input/output works in C/C++/Java/Python.

The problem qaly is an example of how to take multiple lines of input. Usually the first input will be how much more input to expect which you can then loop on: 

#include <iostream>
using namespace std;
int main(){
    int tt;  
    double sum = 0;  
    cin >> tt;   // pipe first stdin to tt
    while (tt--) {  // false when input empty 
        double a, b;  // each loop declare these 2 new assignables
        cin >> a >> b; // take 2 more inputs
        sum = sum + (a * b);  // accumulator
        }
    cout << sum;  // pipe sum to stdout
}

Do enough of these until you get the hang of I/O in your language.

Big-O explained

The first resource we can use where 'running time' is actually explained is the beginning chapters of Functional Data Structures by Prof Ragde at Waterloo try reading the first few chapters up to 2.2.

The formal definition is:

\(f(x) \le c\cdot g(x)\) for all inputs \(x \ge x_0\) as x (inputs) goes to infinity. If that inequality holds then f(x) is in O(g(x)) which represents a worse-case scenario upper bound. A 'best-case' (wishful thinking) scenario for insertion sort is you don't have to do anything but attach the element to the already sorted list and the worst-case is the entire list is in reverse order and every element looked at and moved.

f() function is a cost function derived from a cost model. It's where you count up every constant operation and loop costs by analyzing source code. This is not always easy since what costs do you assign when the code triggers garbage collection or triggers memory allocation because an object has been created deep in some library code.

The g() function is chosen from a list of math functions that represent a 'family of functions'. A family means all functions that are linear, all functions that are quadratic, all functions that are linearithmic etc. It actually is a specific list as we'll soon see.

Why are constants hidden in big-O? If you read SICP their example is 16 and 8 both differ by a factor of 2 and 8 but we don't care which factor. In real life the constant factors do matter if they are hiding a lot of operations. A classic example of this is when Knuth was writing the Stanford GraphBase (contents are in Vol 4A of TAOCP) and his real-world testing of popular graph algorithms found that 'Fibonacci heaps' that were analyzed using a technique called amortized analysis weren't any better than Kruskal's old algorithm because the analysis was hiding large constant operations. One way to reduce constants is to do multiple rounds of a linear algorithm cleaning up the graph because O(n + n) or O(2n) is still O(n).

O(g(x)) represents an upper bound for all functions in the family g(x). It doesn't represent growth since f(x) = x2 does not grow to be quadratic, f(x) already is quadratic, thus f() is in \(O(n^{2})\) or upper bounded by every quadratic function. A function n or f(n) is bounded by n, n2 and nn that's why it's an upper bound not the upper bound as there's many upper bounds but we'd like the closest fitting bound.

If the inputs are supposed to be 'near infinite' then n is used instead of x in f(n) and all algorithm analysis you are generally assuming some kind of infinite inputs but here in the USACO guide we can see that if the input is small then we can get away with some very bad runtimes even exponential.

There is many more like little-o (strictly less than), big-Omega (strictly greater than), or big-Theta \(\theta(n)\) which describes a tight lower and upper bound. For example a program that searches a single list for a specific element is going to be at least a lower bound of O(n) because it has to look at everything once and no more than O(n) in the worst case scenario where the item is not found and the entire list is exhausted so it is \(\theta(n)\).

Shell game

The first problem is shell game. You have to make an account on USACO.org to submit entries here: https://usaco.org/index.php

The sample input first line tells us how many times to loop input. Second line tells us shell 2 and 1 were swapped and the guess was 1. If the pebble was in 1 then the guess is incorrect as it swapped with 2. If the pebble was in 2 then the guess was correct.

Write out some examples of keeping track of the state for all 3 potential initial pebble positions.

  • If pebble init 1 then 2, 3, 1
  • If pebble init 2 then 1, 1, 3
  • If pebble init 3 then 3, 2, 2

We only have to update whenever swap a or swap b equals the current position of any of the 3 initial positions. Here's some scratch work I did 

Init swap array a[0] = 1, a[1] = 2, a[2] = 3
Init guess array g[0] = 0, g[1] = 0, g[2] = 0 
Loop array
 If swap1 = a[i] then a[i] = swap2
 If swap2 = a[i] then a[i] = swap1
 // after state change loop a again
 is guess = a[i]? increase g[i]
Return max of g

This is where I know I'm probably not representing the optimal simulation because I have 2 if-statements and you always want to just do a single operation if you can not a pile of if's. I tried this solution and it worked all tests passed using 1.6mb of memory and 1ms execution time. 

#include <array>
#include <cstdio> // needed for i/o
#include <algorithm> // needed for max

int main() {
  // get the first line of input
 freopen("shell.in", "r", stdin);
 int n;
 scanf("%d", &n);

 std::array<int, 3> state = {1, 2, 3};
 std::array<int, 3> guess{};

 for (int i = 0; i < n; i++) {
     int a, b, g;
     scanf("%d %d %d", &a, &b, &g);
  for (int s = 0; s < 3; s++) {
      if (a == state[s]) {state[s] = b;}
      else if (b == state[s]) {state[s] = a;}}
  for (int x = 0; x < 3; x++) {
      if (g == state[x]) {guess[x]++;}}}

 freopen("shell.out", "w", stdout);
 printf("%d\n", std::max({guess[0], guess[1], guess[2]}));}

Looking at their solution they accurately replicated the simulation while I hacked it. I have 2 nested loops but they only loop 3 times (constant) so my solution is n * c * c or nc which is O(n) as constants are ignored in Big-O. Their solution runs with 2.8mb memory and 2ms execution time so my hacked solution is faster for only 10 tests the server does. If I were to guess why I'm not simulating every swap only updating when the pebble is moved.

You only have to include the file in/out if the problem statement asks for a file otherwise stdout can be used.

Mixing milk

Second problem in USACO guide bronze. As usual write out examples to mimic the state transition but the problem statement has done that for us.

  • some kind of mod(3) index looping so when last bucket reached wraps around to first bucket again.
  • the capacity bucket(next) - bucket(next) milk compared to the existing milk in bucket(n), take their min and adjust
#include <algorithm>
#include <cstdio>
#include <iostream>
#include <vector>

int main() {
    std::vector<std::pair<int, int>> v;
    freopen("mixmilk.in", "r", stdin);

	for (int i = 0; i < 3; i++) {
		std::pair<int, int> bi = std::make_pair(0,0);
		std::cin >> bi.first >> bi.second; 
		v.push_back(bi);
	}
	
	for (int i = 0; i < 100; i++) {
		int n = i % 3;  
		int next = (i+1) % 3;  
		int fill = std::min(v[n].second, v[next].first - v[next].second);
		v[n].second = v[n].second - fill;
		v[next].second = v[next].second + fill;
	}
	freopen("mixmilk.out", "w", stdout);
	std::cout << v[0].second << '\n' 
	          << v[1].second << '\n'
		  << v[2].second;
}

My solution is similar to theirs except I made a pair data structure and stuffed it in a vector, and I used 'magic numbers' that's when you don't assign a constant variable where it only has to be changed in one place and there's just some magic number without documentation in the code. My solution uses 3.3mb memory and executes 2ms. Their solution is 3.4mb of memory and 2ms execution so I somehow beat them again but these are utterly worthless micro optimizations.

Stuck in a Rut

We may as well start with the hardest problem Stuck in a Rut and work backward. This seems like an easy problem if we can identify why 2 cows will intersect. If you write out some examples to find some patterns these 2 cows eventually meet:

  • north 11,2
  • east 10,4
    • initial state: (northx > eastx), (northy < easty).
    • meet in 2 or easty - northy

The first 2 cows in the sample input meet but at the same time:

  • north 5,3
  • east 3,5
    • (northx > eastx), (northy < easty) and (northx = easty), (northy = eastx)

First cow eventually crashes into something let's find out who

  • east 3,5
    • second cow has to be moving northx > 3 and northy < 5
  • north 11,2 or north 8,1
    • smaller northx hits the east 3,5 cow first
    • meet in 5 or northx - eastx

Second cow eventually intersects:

  • north 5,3
    • other cow had to be moving eastx < 5, and easty > 3
  • east 4,6 or east 3,5 (equal so discarded)
    • bigger eastx intersects first
    • meet in 3 or easty - northy

Compare all the north traveling cows compared to the east traveling cows, eliminating if (a,b = b,a), seeing if there is a pattern to the subtraction to find the exact meet, and whatever is left over test if it's movement is on the same y or x-axis, if not return infinity.

Gitless

Now we are going to take all the above introductions to conceptual design and rewrite the git interface to make it intuitive to use. We have to learn git anyway so may as well write a CLI (command line interface) or you could make it ridiculous and call it 'visual git' or something with DAG graphics.

Many companies today use a customized Mercurial fork or rewritten clone instead of git like Meta's rust written EdenFS and Sapling SCM because git was originally developed to take advantage of linux file system optimizations ergo is conceptually attached to a file system while Mercurial can be easily adapted to be stuffed in a database, distributed or run on any OS. In a Mercurial style of version control, you probably don't care about individual commits so you don't even add comments to commits you instead care about entire branch development.

Let's learn what git actually is by watching MIT's version control lecture from it's course The Missing Semester of Your CS Education. Lecture starts ~7m in. So far the data structure and model is trivial to understand if you did any intro course here such as CS19 learnings about DAGs.
