Strategy

Find a mentor. There is someone somewhere doing what you want to do and making a lot of money doing it and your task is to do enough material here to reach a level where you can get in with them somehow and absorb everything. Once you are in all you have to do is be the reliable person they turn to whenever some (tedious at first) work needs to be done and if this work over time becomes increasingly more skilled then you have found the right mentor and you are set for life. If they get promoted, you get promoted. If they found a company, you are the first hire. If they get hired at a top tier trading firm everyone says you have no chance of getting into guess who they will tell the hiring managers they should also hire.

You approach this person and sell yourself telling them you are available if they have some work to offload because you want to learn it all from the ground up. There is always something to do and your goal is to be this magic portal where work they despise doing can be dumped and it returns as per specifications they gave you then there's time for them to show you things. You make your way up to highly sophisticated work until you end up being essential because it's a mutually beneficial relationship. I actually took all the above advice from a comedian many years ago and it worked. In finance this can be tricky because of signing multiple NDAs even between employees but if you sell yourself to them that you are serious about being mentored they will tell management to make it happen.

Easiest way to meet a mentor is get the lowest job you can at a place where there's experts making the place money and then approach them asking for work to do on top of whatever else you're doing. After a few months of this if any work comes up in their department you're the one they want to move in.

USACO guide

In competitive programming you submit source code as a solution to a problem and the online judge compiles and runs your code against (often 'secret') tests. There is a huge list of online judges available like Kattis, SPOJ, Timus, Aizu, and more worldwide.

First task is to choose one of these languages. We'll learn C++ for USACO so pick something else here that you want to use. List all the problems by easiest and start with the very first one 'Velkomin!'. You get unlimited submissions to the Kattis judge.

The point of doing this is to learn algorithm design which really is 'system design' or 'platform engineering' or 'high throughput data pipelines' it's all the same there's some complex things that need to be optimized as much as possible. This is the bare minimum you need to step to some mentor somewhere and solve their problem like their trading data they use for training has some crazy garbage where data was included after trades ended that day but aren't accurately timestamped like there's a timestamp discrepancy in numerous feeds so you need a fast way to complete search and label over 100 terabytes of data discarding everything that should not be included in the training set. Now imagine 80+ of these data pipelines all dumping terabytes on you every minute so your algorithm needs to be as modular as possible so you can abstract it over any kind of noisy data and it needs to run automatically 24/7 as data comes firing in all hours globally. The faster you do this the more money everyone you work with (including you) makes because you are a race car programmer now and this data is the pit stop between trades where the first wins.

This is why we are learning this.

'Fast' languages

There is no such thing as an interpreted language there is only languages which don't have a good compiler. C++ is only as fast as there is memory allocations in the code so while you may have heard it's the 'fastest' it's not if you don't know what you're doing in fact most in the wild C++ code is junk that runs just as slow as whatever you can think of. We used to be taught static arrays were the best possible data structures but they rely on magic hardware adders which is useless for parallel computing which we are now doing for absolutely everything where I am anyway so the preferred structure is trees now.

Being fast means you have an intimate relationship with how your preferred language works collecting garbage or allocating memory and objects so we need to learn how this works and thankfully there is a guide for this called plai.org in other words choose the language you best like and never choose something someone claims is 'fast' because you can probably tune your preferred language to obliterate most C++. Common Lisp you can go right down to the compiler and generate assembly (same with Julia, SML, OCaml) so sky is the limit.

Input/Output

Read the book or online documemtation of the language you chose or use the USACO guide first chapter to teach yourself how input/output works in C/C++/Java/Python. There's the USACO online IDE too if you want.

The problem qaly is an example of how to take multiple lines of input. Many of these problems the first input will be how much more input to expect which you can then loop on: 

#include <iostream>
using namespace std;
int main(){
    int tt;  
    double sum = 0;  
    cin >> tt;   // pipe first stdin to tt
    while (tt--) {  // false when input empty 
        double a, b;  // each loop declare these 2 new assignables
        cin >> a >> b; // take 2 more inputs
        sum = sum + (a * b);  // accumulator
        }
    cout << sum;  // pipe sum to stdout
}

Finish up to 1.4 Easy problems on Kattis or until you get how I/O works in your language.

Big-O explained

The first resource we can use where 'running time' is actually explained is the beginning chapters of Functional Data Structures by Prof Ragde at Waterloo try reading the first few chapters up to 2.2.

The formal definition is:

\(f(x) \le c\cdot g(x)\) for all inputs \(x \ge x_0\) as x goes to infinity. If that inequality holds then f(x) is in O(g(x)).

Full explanation:

  • x floating around by itself or f(x) = x or y = x represent the same thing.
  • the big-O definition doesn't care if c = 1 or c = 1 trillion.
  • if inputs 'go to infinity' traditionally you write n instead of x.
  • x0 means some specific input of x or threshold.
    • can be the first input or the trillionth input the big-O definition doesn't care.
    • all inputs are positive as it would make no sense to model negative input.
  • f(x) is a cost function.
  • g(x) is chosen from a list of existing math functions, see below.

Toy example: Suppose a program consists of 2 constant operations followed by a single loop with 10 constant operations in the loop body. In the RAM model this is f(x) = 10x + 2

  • f(n) = 10n + 2
    • n because f(x) 'goes to infinity'
  • 10n + 2 <= cn
    • true if c = 11 for all inputs x >= 2
    • also true if c = 12, a thousand, a billion

f() function is a cost function derived from a cost model. It's where you count up every constant operation and loop costs by analyzing source code. This is not always easy since what costs do you assign when the code triggers garbage collection or something triggers memory allocation because an object has been created deep in some library code or for functions that can consume other functions as inputs.

The g() function is chosen from a list of math functions that represent a 'family of functions'. A family means all functions that are linear, all functions that are quadratic, all functions that are linearithmic etc. It actually is a specific list as we'll soon see.

Why are constants hidden in big-O? If you read SICP their example is 16 and 8 both differ by a factor of 2 and 8 but we don't care which factor. In real life however you will never have infinite inputs so the constant factors do actually matter if they are very large hiding a lot of operations.

O(g(x)) represents an upper bound for all functions in the family g(x). It doesn't represent growth since f(n) = n2 does not grow to be quadratic, f() already is quadratic, thus f() is in \(O(n^{2})\) or upper bounded by every quadratic function. The function n is bounded by n, n2 and nn that's why it's an upper bound not the upper bound as there's many upper bounds but you obviously want the closest fitting bound.

Big-O is in a hierarchy of asymptotics there's many more like little-o (strictly less than), big-Omega (strictly greater than), or big-Theta which describes a lower bound and an upper bound. For example an algorithm that searches a list is going to be at least a lower bound of O(n) because it has to look at everything once and upper bounded by O(n) so it is \(\theta(n)\).
